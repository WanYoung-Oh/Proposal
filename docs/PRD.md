## 1. 제품 개요

### 1.1 한 줄 정의

공공정보화 사업 **RFP(제안요청서)** 를 입력하면, **제안 PM**과 AI가 단계별로 협업하여 **제안전략 7단계**를 완주하고, **최종 전략 문서·제안서 스토리보드**를 만들어 주는 내부용 AI 서비스.

### 1.2 제품이 해결하는 문제

| 현재 Pain Point | 본 제품의 해결 방향 |
|----------------|-------------------|
| RFP 분석에 PM·실무자가 수일 소요 | RFP PDF 업로드 → 사업개요·요구사항·평가항목 **자동 추출** |
| 현장 인터뷰·영업 정보가 문서화되지 않음 | PM 입력 → AI가 **구조화된 요구사항·경쟁력 분석**으로 정리 |
| 과거 제안서·방법론 활용이 PM 개인 역량에 의존 | **26건+ 유사 사례 RAG** 참조로 차별화·수행전략 초안 생성 |
| 좋은 슬라이드 사례 찾기가 수동·비효율 | **주제 기반 슬라이드 검색** + 선정 사유 설명 |

### 1.3 핵심 가치

1. **속도** — RFP 분석·전략 초안 작성 시간 단축  
2. **일관성** — 회사 제안전략 방법론 7단계를 매번 동일하게 적용  
3. **지식 재활용** — 과거 수주·실주 사례와 방법론을 실시간 참조  
4. **PM 주도** — AI는 보조, 최종 판단·수정은 PM이 Human-in-the-Loop로 수행

## 3. 사용자 및 페르소나

### 3.1 주요 사용자

| 페르소나 | 역할 | 주요 니즈 |
|---------|------|----------|
| **제안 PM** | 제안전략 수립 주 담당 | RFP 빠른 이해, 차별화·수행전략 초안, 스토리보드 |

- **동시 사용자:** 1인 인터랙티브 (MVP~1차 릴리스)
- **운영 장비:** Mac Mini M5 Pro (상시)
- **접근 방식:** 내부망 Streamlit → 이후 Next.js 웹 UI

## 4. 제품 범위

### 4.1 In Scope (포함)

| 영역 | 내용 |
|------|------|
| RFP 분석 | PDF 업로드 → 사업개요·공식 요구사항·평가항목 자동 추출 |
| PM 협업 | 비공식 요구사항·경쟁력 분석·(선택) 의사결정 입력 |
| AI 전략 생성 | 경쟁우위 차별화(5단계)·사업수행전략(7단계) |
| 지식베이스 | 제안서 26건(목표 43건) + 방법론 3건 RAG |
| 슬라이드 검색 | 주제별 유사 슬라이드 이미지 + 선정 사유 |
| 산출물 | Markdown 다운로드 (MVP) |
| LLM 선택 | sLLM(Qwen3.5) + LLM API(Solar/Claude) **노드별 분리** — STEP 5·7은 API 권장 |

### 4.2 Out of Scope (제외 — MVP)

| 항목 | 사유 |
|------|------|
| 제안서 PPT 자동 작성 | 스토리보드·전략 문서까지만 |
| 다중 사용자·권한 관리 | 1인 사용 전제 |
| RFP HWP 직접 업로드 | PDF 변환 후 업로드 |
| PDF 최종 산출물 | 향후 릴리스에서 검토 (MVP는 Markdown만) |
| 클라우드 SaaS 배포 | Mac Mini 로컬 운영 |
| 실시간 나라장터 연동 | RFP는 수동 수집 |

## 5. 핵심 사용자 여정

```
[시작] PM이 새 RFP PDF 업로드
    ↓
[자동] AI가 STEP 1~3 분석 (사업개요·공식 요구·평가항목)
    ↓ PM 검토·수정
[입력] PM이 STEP 2 비공식 요구·STEP 4 경쟁력 분석 입력
    ↓
[AI] 유사 사례 RAG 참조 → STEP 5 차별화 전략 생성
    ↓
[선택] STEP 6 의사결정 — 입력 또는 건너뛰기
    ↓
[AI] STEP 7 수행전략·스토리보드 생성
    ↓
[완료] Markdown 다운로드

(별도) PM이 "재해복구 전략" 등 주제로 슬라이드 샘플 검색 → 참고 슬라이드 3장 확인
```

## 7. 7단계 프로세스 — 단계별 산출물

> 방법론 출처: `docs/ref/process.pdf`

| 단계 | 활동 | 담당 | 데이터 소스 | 시스템 산출물 | 필수 |
|------|------|------|------------|--------------|------|
| 1 | 사업개요 | 제안PM | RFP 자동 | 사업명·발주기관·범위·경쟁사 | M |
| 2a | 고객 요구 (공식) | 제안PM | RFP 자동 | 제안요청사항·차별화 포인트 | M |
| 2b | 고객 요구 (비공식) | PM+영업 | PM 입력 | Hidden Needs·Pain Point·쟁점 | M |
| 3 | 평가항목 분석 | 제안PM | RFP 자동 | 배점·특징·**high_score_items**(고배점 요구) | M |
| 4 | 경쟁력 분석 | 제안PM | PM 입력 | 장단점 비교표 | M |
| 5 | 경쟁우위 차별화 | 제안PM | 2+3+4+RAG | 5-1 경쟁구도(약점만회·강점부각) / 5-2 핵심이슈(고배점·Hidden·Pain) | M |
| 6 | 의사결정 사항 | PM+부장 | PM+AI | 의사결정 항목·권고안 | O |
| 7 | 사업수행전략 | 제안PM | 1~6 종합 | CSF·요약·**MECE 4영역** 이행방안·스토리보드 | M |

### 7.1 단계별 PM 개입 방식

| 유형 | 해당 단계 | PM 행동 |
|------|----------|---------|
| **자동** | 1, 2a, 3 | RFP 업로드 → AI 추출 → PM 검토·수정 |
| **입력** | 2b, 4, 6 | **폼 입력**(MVP) → AI 구조화 (6은 skip 가능) |
| **AI 도출** | 5, 7 | STEP 5: RAG 참조 → 전략 생성 / STEP 7: 누적 정보 + STEP 5 RAG 재활용 |

### 7.2 STEP 5·7 산출물 구조 (PLAN §10과 동일)

**STEP 5 — 경쟁우위 차별화**
- **5-1 경쟁구도:** 경쟁사 대비 (1) 약점 **만회** (2) 강점 **부각** — 평가항목·RFP 연계
- **5-2 핵심이슈:** (1) **고배점** 평가 요구 (2) **Hidden Needs** (3) **Pain Point** — 이슈별 우리 차별화 방안

**STEP 7 — 사업수행전략**
- **7-1 CSF:** 핵심성공요소 5~7개 (STEP 5와 직접 연결)
- **7-2 수행전략 요약:** 경영진 보고용 1페이지 분량, 핵심 메시지 3~5개
- **7-3 MECE 4영역 이행방안:** A.경쟁구도·포지셔닝 → B.기술·솔루션 → C.프로젝트관리·수행 → D.고객·이해관계자
- **7-4 제안서 간이 스토리보드:** 목차별 핵심메시지·차별화요소·연결 CSF

### 7.3 최종 Markdown 문서 구조 (문서 4대 섹션)

> STEP 7-3 **MECE 4영역**(이행방안 내용)과 **별개** — `format_output`이 생성하는 **통합 문서 목차**이다. PLAN §14 Week 2 참고.

| 섹션 | 포함 단계 | 내용 |
|------|----------|------|
| ① 고객 분석 | STEP 1~3 | 사업개요·공식/비공식 요구·평가항목 |
| ② 경쟁 포지션 | STEP 4 | 경쟁력 비교표 |
| ③ 차별화 전략 | STEP 5~6 | 경쟁구도·핵심이슈 차별화·의사결정 |
| ④ 사업수행전략 | STEP 7 | CSF·요약·MECE 4영역 이행방안·스토리보드 |

| 과거 제안서 | **26건** ✅ | 43건 | RAG 유사 사례 |
| 제안전략 방법론 | **3건** ✅ | 3건 | RAG 방법론 |
| RFP (과거 사업) | **9/26** ⚠️ | 26건+ | tags·평가항목 추출 |
| meta.json | 26건 ✅ | — | 연도·기관·수주/실주·규모 |
| tags.json | 26건 ✅ | — | 전략 키워드·평가항목·차별화·**전략 요약(strategy_summary)** |

### 8.2 데이터 품질 요구

- 신규 제안서 추가 시 `project_id` 형식: `{연도}_{기관코드}_{사업명}`
- RFP 미보유 프로젝트는 RAG 품질 저하 가능 → P1 RFP 수집 지속
- `tags.json`의 `strategy_keywords`·`evaluation_criteria`는 검색 필터·프롬프트 보강에 활용
- `tags.json`의 `strategy_summary`는 RAG 검색 결과에 포함되어 STEP 5 프롬프트에 **프로젝트 수준 전략 요지**로 주입 — 슬라이드 청크가 담지 못하는 전략 방향 맥락 보완

### 8.3 초기 빌드 vs 운영

| 작업 | 장비 | 빈도 |
|------|------|------|
| 임베딩·Qdrant 인덱싱·PNG 렌더링 | RTX 3090 → Mac Qdrant 업로드 | 최초 1회 + 데이터 추가 시 |
| RAG 쿼리·LLM·UI | Mac Mini | **상시** |

## 14. 용어 정의

| 용어 | 설명 |
|------|------|
| RFP | 제안요청서 (Request for Proposal) |
| PM | 제안 프로젝트 매니저 (제안 담당자) |
| RAG | 검색 증강 생성 — 과거 문서에서 유사 내용 검색 후 LLM에 제공 |
| CSF | Critical Success Factor, 핵심성공요소 |
| Human-in-the-Loop | AI가 멈추고 사람(PM) 입력을 기다린 후 재개하는 방식 |
| 스토리보드 | 제안서 목차별 핵심 메시지·차별화 포인트 초안 |
| sLLM | Small LLM — Mac Mini 로컬 Qwen3.5-9B(MLX). RFP 추출·PM 입력 구조화·슬라이드 사유 |
| LLM API | Solar Pro / Claude — STEP 5·7 전략 추론에 권장 |
| MECE | Mutually Exclusive, Collectively Exhaustive — STEP 7-3 **이행방안** 4영역(A~D) 구조 |
| 문서 4대 섹션 | 최종 Markdown **목차** 구조 — ①고객분석 ②경쟁포지션 ③차별화 ④수행전략 (§7.3) |
